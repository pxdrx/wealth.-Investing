# Relatório de Auditoria: Compatibilidade Código × Schema Supabase

**Data:** 2025  
**Escopo:** Fase de auditoria de compatibilidade real entre código e schema/policies do Supabase.  
**Regra:** Nenhum arquivo foi alterado; apenas relatório técnico.

---

## 1. Resumo executivo

### 1.1 O que foi auditado

- **Código:** Uso de Supabase nos arquivos:
  - `app/api/journal/import-mt5/route.ts`
  - `lib/prop-stats.ts`
  - `lib/accounts.ts`
  - `lib/bootstrap/ensureDefaultAccounts.ts`
  - `app/app/journal/page.tsx`
  - `app/app/page.tsx` (Dashboard)
  - `app/app/wallet/page.tsx`
  - `lib/profile.ts`
- **Repositório:** Busca por migrações, SQL ou definição de schema (pastas `supabase/`, `**/migration*`, `*.sql`). **Nenhum arquivo de schema encontrado.**
- **Banco real:** **Não foi possível validar.** Não há acesso ao projeto Supabase a partir deste ambiente; não existem dumps de schema ou migrações versionadas no repo. Portanto, **todas as afirmações sobre o estado real do banco (colunas, tipos, RLS, policies) estão marcadas como não confirmadas**.

### 1.2 Metodologia

- Extração do **contrato esperado pelo código**: para cada tabela, listagem de colunas usadas em `select`, `insert`, `upsert`, e em filtros `.eq()`.
- Identificação de **premissas implícitas**: unique keys, colunas geradas, nullability, FK.
- Listagem de **riscos de divergência**: o que quebra ou se comporta mal se o schema real não bater com esse contrato.
- **Severidade** por item (alta/média/baixa), **causa provável**, **impacto** e **correção recomendada**.

### 2.1 `profiles`

|-------|----------|----------------------|
| `lib/profile.ts` | SELECT | `id`, `display_name` |

**Premissas implícitas no código:**

- Tabela `public.profiles` existe.
- `id` é PK e coincide com `auth.users.id` (UUID).
- `display_name` pode ser NULL (código trata `profile?.display_name?.trim()`).
- Upsert por `id` exige unique/primary key em `id`.

**O que validar no banco (não confirmado):**

- Colunas: `id` (UUID, PK?), `display_name` (text?, nullable?).
- RLS habilitada? Policies: SELECT/UPDATE/INSERT para usuário autenticado em `id = auth.uid()`.
- Se RLS bloquear SELECT em `profiles`, login/onboarding quebram.

| Fonte | Operação | Colunas / Observação |

| `lib/accounts.ts` | SELECT | `account_id`, `firm_name`, `phase`, `starting_balance_usd`, `profit_target_percent`, `max_daily_loss_percent`, `max_overall_loss_percent`, `reset_timezone`, `reset_rule`; filtro `account_id` IN (...) |

**Premissas implícitas:**

- **`user_id`:** usado em import-mt5 e prop-stats. Se não existir no banco, a query falha (código já trata erro e retorna 500 com mensagem).

- Se `user_id` não existir: já identificado como risco; código hoje falha com 500 e mensagem.
- RLS: policies devem permitir SELECT/INSERT para o dono (por `user_id` ou por join com `accounts.user_id`); caso contrário, bootstrap e import quebram.

### 2.4 `journal_trades`

| Fonte | Operação | Colunas / Observação |

## 3. Divergências encontradas (inferidas / riscos)

Riscos são **inferidos** a partir do contrato do código. O banco real **não foi consultado**; portanto, “divergência” aqui significa “o que quebra ou fica inconsistente **se** o schema não estiver como o código espera”.

---

### D1. `prop_accounts.user_id` ausente ou não preenchido

- **Severidade:** **Alta**
- **Descrição:** Código filtra `prop_accounts` por `user_id` em import-mt5 e prop-stats. Bootstrap insere `user_id` em `prop_accounts`.
- **Causa provável:** Schema antigo sem coluna `user_id`; ou RLS/trigger que não preenche `user_id`.
- **Impacto:** Import MT5 para conta prop retorna 500 (já tratado); prop-stats retorna vazio; bootstrap pode falhar no insert se a coluna for NOT NULL e não for enviada, ou falhar se a coluna não existir.
- **Correção recomendada:** No banco: adicionar coluna `user_id` (UUID, FK para auth.users) se não existir; garantir que seja preenchida no insert (bootstrap já envia). No código: já há tratamento de erro com mensagem clara; manter.
- **Ordem:** 1 (crítico para import e prop dashboard).

---

### D2. `journal_trades.net_pnl_usd` não gerado no banco

- **Severidade:** **Média**
- **Descrição:** Código não envia `net_pnl_usd` no insert e comenta que é “generated by the DB”. Se não houver generated column nem trigger, a coluna fica NULL.
- **Causa provável:** Schema criado sem GENERATED COLUMN / trigger para `net_pnl_usd`.
- **Impacto:** Prop-stats e Dashboard usam `net_pnl_usd` com fallback para 0 ou para `pnl_usd + fees_usd` quando NULL; funcionalmente não quebra, mas dados históricos podem ficar inconsistentes se em algum momento o app escrever `net_pnl_usd` manualmente ou se o cálculo no banco for diferente do fallback.
- **Correção recomendada:** No banco: definir `net_pnl_usd` como GENERATED ALWAYS AS (pnl_usd + fees_usd) ou trigger que preenche; ou garantir que o código sempre envie `net_pnl_usd` no insert (remover premissa de “generated”). Documentar no código a decisão (gerado vs enviado).
- **Ordem:** 2.

---

### D3. Unique em `journal_trades` (user_id, account_id, external_source, external_id)

- **Severidade:** **Alta**
- **Descrição:** A deduplicação de reimport depende de no máximo uma linha por (user_id, account_id, external_source, external_id). O código faz SELECT antes de insert, mas não há garantia no banco.
- **Causa provável:** Constraint unique não criada.
- **Impacto:** Race condition ou bug pode inserir duplicatas; contagens e relatórios ficam errados; possível inconsistência com prop cycle (soma de trades duplicados).
- **Correção recomendada:** No banco: criar UNIQUE (user_id, account_id, external_source, external_id). Código já trata 23505 como duplicata no insert; manter.
- **Ordem:** 1.

---

### D4. RLS / policies bloqueando operações

- **Severidade:** **Alta**
- **Descrição:** Se RLS estiver habilitada sem policies corretas, SELECT/INSERT podem retornar vazio ou falhar.
- **Causa provável:** Policies ausentes ou usando condição diferente da que o app usa (ex.: `auth.uid()` vs `user_id`).
- **Impacto:** Login/onboarding (profiles); listagem de contas e import (accounts, prop_accounts); journal e dashboard (journal_trades); payouts e wallet (prop_payouts, wallet_transactions); logs (ingestion_logs).
- **Correção recomendada:** Para cada tabela, confirmar RLS e policies (SELECT/INSERT/UPDATE conforme uso). Garantir que usuário autenticado acesse apenas linhas com `user_id = auth.uid()` (ou equivalente via join).
- **Ordem:** 1.

---

### D5. `profiles`: upsert com `onConflict: "id"`

- **Severidade:** **Média**
- **Descrição:** Upsert usa `onConflict: "id"`. Exige unique ou primary key em `id`.
- **Causa provável:** PK em `id` é padrão; se por algum motivo não existir, upsert pode falhar ou se comportar de forma inesperada.
- **Impacto:** Onboarding não consegue salvar display_name.
- **Correção recomendada:** Garantir que `profiles.id` seja PK (ou unique). Confirmar no Supabase.
- **Ordem:** 2.

---

### D6. `ingestion_logs.meta` tipo e nullability

- **Severidade:** **Média**
- **Descrição:** Código envia `meta` como objeto. Se a coluna for tipo incompatível (ex.: text) ou não existir, insert falha.
- **Causa provável:** Coluna `meta` como text ou ausente.
- **Impacto:** Log de import (ok/erro) falha; diagnóstico fica mais difícil; o restante do import já terá sido commitado (logs são ao final).
- **Correção recomendada:** Garantir coluna `meta` jsonb (ou json) nullable. Se for obrigatória, código deve sempre enviar pelo menos `{}`.
- **Ordem:** 2.

---

### D7. `accounts.created_at` e ordenação

- **Severidade:** **Baixa**
- **Descrição:** Código ordena por `created_at` em `lib/accounts.ts`. Se a coluna não existir, a query falha.
- **Causa provável:** Coluna omitida na criação da tabela.
- **Impacto:** Listagem de contas quebra; header e fluxos que dependem de contas quebram.
- **Correção recomendada:** Garantir coluna `created_at` (timestamptz, default now()). Confirmar no banco.
- **Ordem:** 3.

---

### D8. `prop_accounts` – conjunto de colunas do insert (bootstrap)

- **Severidade:** **Média**
- **Descrição:** Bootstrap insere `user_id`, `account_id`, `firm_name`, `phase`, `starting_balance_usd`, `profit_target_percent`, `max_daily_loss_percent`, `max_overall_loss_percent`, `reset_timezone`, `reset_rule`. Se alguma coluna obrigatória faltar no banco ou tiver nome diferente, insert falha.
- **Causa provável:** Schema com colunas diferentes ou adicionais obrigatórias.
- **Impacto:** Bootstrap falha; usuário novo não recebe contas padrão; possível impacto em listagem e import se contas prop não existirem.
- **Correção recomendada:** Alinhar schema de `prop_accounts` à lista acima; garantir que não haja NOT NULL sem default para colunas não enviadas.
- **Ordem:** 2.

---

### D9. Tipos incompatíveis (timestamps, números, enums)

- **Severidade:** **Média** (conjunto)
- **Descrição:** Código envia strings ISO para `opened_at`/`closed_at`/`paid_at`; números para valores USD e percentuais; strings para `phase`, `kind`, `tx_type`, `status`, `source`. Se o banco esperar tipos diferentes (ex.: enum, date sem time), pode haver erro ou coerção silenciosa.
- **Causa provável:** Definição de colunas com tipo diferente do uso (ex.: date vs timestamptz).
- **Impacto:** Erros de insert/update ou dados incorretos (ex.: timezone).
- **Correção recomendada:** Confirmar tipos no Supabase (timestamptz para datas; numeric/real para valores; text para enums ou usar enums nativos). Ajustar código ou schema conforme necessário.
- **Ordem:** 2.

## 5. Causa provável, impacto e correção (resumo)

- **D1:** Schema antigo ou migração incompleta → import/prop-stats falham ou retornam vazio → adicionar/garantir `user_id` em `prop_accounts` e preencher.
- **D2:** Falta de generated/trigger para `net_pnl_usd` → NULL em novos trades; app usa fallback → definir generated/trigger ou passar a enviar `net_pnl_usd` no insert e documentar.
- **D3:** Falta de unique → risco de duplicatas em reimport → criar UNIQUE (user_id, account_id, external_source, external_id).
- **D4:** RLS sem policies adequadas → acesso negado em várias rotas → revisar policies por tabela (profiles, accounts, prop_accounts, journal_trades, prop_payouts, wallet_transactions, ingestion_logs).
- **D5:** Falta de PK/unique em `profiles.id` → upsert falha → garantir PK em `profiles.id`.
- **D6:** `meta` inexistente ou tipo errado → insert em ingestion_logs falha → garantir `meta` jsonb nullable (ou ajustar código).
- **D7:** `accounts.created_at` ausente → query de accounts falha → adicionar coluna com default.
- **D8:** Colunas obrigatórias diferentes em `prop_accounts` → insert do bootstrap falha → alinhar schema ao insert.
- **D9:** Tipos diferentes (datas, números, enums) → erros ou dados errados → alinhar tipos no banco ao uso no código.

1. **Confirmar schema real** no Supabase (SQL ou Dashboard): listar colunas, tipos, constraints, RLS e policies para as 7 tabelas.

3. **prop_accounts.user_id (D1):** existência e preenchimento; já há tratamento de erro no código.
4. **journal_trades unique (D3):** criar constraint para deduplicação segura.
